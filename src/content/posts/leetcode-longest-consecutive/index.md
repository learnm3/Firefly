> 这道题的 O(n) 解法有一个微妙但致命的直觉——「只在序列的起点开始计数」。游戏里的连击系统和签到系统，本质上就是同一道题。

---

## 题目回顾

[LeetCode 128. Longest Consecutive Sequence](https://leetcode.cn/problems/longest-consecutive-sequence/) — 难度：中等

**给定一个未排序的整数数组，找出数字连续的最长序列的长度。要求 O(n) 时间复杂度。**

```
输入: nums = [100, 4, 200, 1, 3, 2]
输出: 4
解释: 最长连续序列是 [1, 2, 3, 4]，长度为 4
```

关键约束：**不要求序列元素在原数组中连续**，所以可以重组。但要 O(n)——排序会超限。

## 游戏场景：哪里需要找「连续」？

### 场景一：每日签到系统——最长连续登录天数

这是最直白的映射，几乎所有手游/网游都有。

> 玩家的登录记录是一组时间戳（毫秒），转化成「一年中的第几天」后，求最长连续登录天数。

```
登录记录（转换后）: [4, 5, 6, 10, 11, 15, 16, 17, 18]
最长连续登录: 5 → 天 15~18 + 天 4~6 → 等等，让我重算
最长连续登录: 4 → 天 [15, 16, 17, 18] = 4 天
```

```cpp
// 签到系统核心：最长连续登录天数
int LongestLoginStreak(vector<int>& loginDays) {
    unordered_set<int> s(loginDays.begin(), loginDays.end());
    int maxStreak = 0;
    for (int day : s) {
        // 关键：只在连续段的「起点」开始计数
        if (s.find(day - 1) != s.end()) continue;
        int end = day;
        while (s.find(end + 1) != s.end()) end++;
        maxStreak = max(maxStreak, end - day + 1);
    }
    return maxStreak;
}
```

**面试加分点**：签到系统的难点不只是算法——你还需要处理跨月/跨年边界、时区转换、以及「补签卡」对连续性的影响。主动提这些说明你做过真实项目。

### 场景二：连击计数器——动作/格斗游戏

> 鬼泣 / 猎天使魔女 / 怪物猎人中，连击评级是一个离散的等级系统：D(1) → C(2) → B(3) → A(4) → S(5) → SS(6) → SSS(7)。

当玩家在一场战斗中打出了评级序列 `[5, 2, 3, 6, 4, 7, 1]`，问：「这场战斗中最长的一段**连续不降级**的连击跨度是多少？」

如果玩家打出了评级 1 到 7 各一次（不论顺序），那整套评级链都被覆盖过一次，最大连续跨度就是 7——说明玩家从最基础的 D 一路打到了 SSS。

```python
# 连击评级覆盖分析
ratings = [5, 2, 3, 6, 4, 7, 1]  # S, C, B, SS, A, SSS, D
# 是否存在从 D(1) 到 SSS(7) 的完整链？→ 最长连续序列 = 7
# 结论：玩家覆盖了全部 7 个评级，连击掌控力极强
```

这个分析在格斗游戏的 **Training Mode 数据复盘** 中很实用——Mortal Kombat / 街霸的职业选手会关注自己覆盖了多少个连击段位。

### 场景三：装备强化连续等级

> MMORPG 中，装备强化从 +0 到 +20。如果玩家同时拥有 +12、+13、+14、+15 四件不同部位的装备，这四件装备的强化等级构成了一个长度为 4 的连续序列。

某些游戏会为「连续强化等级」提供套装加成——比如你拥有连续 5 个等级的强化装备，额外获得 5% 属性。这就是求最长连续序列。

```cpp
// 装备强化连续等级加成判定
int MaxConsecutiveEnhanceLevels(vector<int>& enhanceLevels) {
    unordered_set<int> s(enhanceLevels.begin(), enhanceLevels.end());
    int maxLen = 0;
    for (int lv : s) {
        if (s.find(lv - 1) != s.end()) continue; // 不是起点，跳过
        int cur = lv;
        while (s.find(cur + 1) != s.end()) cur++;
        maxLen = max(maxLen, cur - lv + 1);
    }
    return maxLen;
}
```

### 场景四：开放世界地图连续区域探索

> 开放世界（旷野之息 / 原神 / 艾尔登法环）中，地图被划分成若干区块（Chunk），每个区块有一个 ID。玩家已探索的区块 ID 为 `[12, 7, 9, 8, 11, 10, 5]`。

最长连续区块 ID 序列是 `[7, 8, 9, 10, 11, 12]`，长度为 6——说明玩家在 ID 7~12 这片区域几乎探索完毕，可以触发「区域完全探索」成就。

对于大地图 LOD 系统，连续区块还可以被合批加载，减少 Draw Call。

### 场景五：音游长连判定

> 节奏游戏（OSU! / 太鼓达人 / Muse Dash）中，perfect/hit 判定被记录为连续的帧号。如果玩家在一段区间内每帧都是 perfect，那这些帧号构成了一个连续序列。最长连续 perfect 帧数 = 最长连续序列。

## 解法分析

### 为什么不能排序？

排序再扫描听起来很自然——O(n log n)，代码三行：

```cpp
// 排序法：O(n log n) —— 不符合题目 O(n) 要求，但面试一定要说出来
sort(nums.begin(), nums.end());
int maxLen = 0, cur = 1;
for (int i = 1; i < nums.size(); i++) {
    if (nums[i] == nums[i-1]) continue;      // 跳过重复
    if (nums[i] == nums[i-1] + 1) cur++;
    else { maxLen = max(maxLen, cur); cur = 1; }
}
return max(maxLen, cur);
```

在游戏场景中，如果签到数据量只有几万条，排序完全可行。但当数据量上到百万级（比如记录每个玩家的每次操作帧号），O(n log n) 和 O(n) 的差距就很明显了。

### O(n) 解法：哈希集合 + 「只在起点开始数」

核心洞察：**如果一个数字 `x` 是某个连续序列的一部分，且 `x-1` 也存在于数组中，那 `x` 不可能是这个序列的起点。我们只需要从「没有前驱」的数字开始向后数。**

```
数组: [100, 4, 200, 1, 3, 2]

放入哈希集合: {100, 4, 200, 1, 3, 2}

遍历:
  x=100 → 99 不在集合中 → 起点！→ 数: 100 → 101? 不在 → 长度=1
  x=4   → 3  在集合中   → 跳过（不是起点）
  x=200 → 199 不在集合中 → 起点！→ 数: 200 → 201? 不在 → 长度=1
  x=1   → 0  不在集合中 → 起点！→ 数: 1→2→3→4→5? 不在 → 长度=4 ✓
  x=3   → 2  在集合中   → 跳过
  x=2   → 1  在集合中   → 跳过

最长 = 4
```

```cpp
int longestConsecutive(vector<int>& nums) {
    if (nums.empty()) return 0;
    
    unordered_set<int> s(nums.begin(), nums.end());
    int ans = 0;
    
    for (int x : s) {
        // 只有「序列起点」才开始计数
        if (s.find(x - 1) != s.end()) continue;
        
        int cur = x;
        while (s.find(cur + 1) != s.end()) cur++;
        ans = max(ans, cur - x + 1);
    }
    return ans;
}
```

### 复杂度分析——为什么是 O(n)？

一眼看上去有嵌套 while 循环，像 O(n²)。但关键点在于：

> **每个数字最多被访问两次**——一次在 for 循环中被遍历到，一次在某个数字的 while 扩展中被 `find` 到。而且 `x-1` 存在的数字根本不触发 while，所以每个元素最多被扩展一次。

| 维度 | 分析 |
|---|---|
| 时间复杂度 | O(n) —— 每个元素最多访问 2 次 |
| 空间复杂度 | O(n) —— 哈希集合存储所有元素 |
| 核心技巧 | **跳过非起点**——if `x-1` exists → skip → 每个元素只被自己的「起点」遍历一次 |

### 为什么这个直觉不容易想到？

因为我们习惯了对数组**从前到后扫描**。但这道题需要我们跳出来思考：「一个序列的起点有什么特征？」——起点前面没有数字。

这是一种**状态判定思维**而非遍历思维。游戏开发中很常见——比如「一个连击的起始帧」是玩家按下攻击键且上一帧不在攻击状态中的那一帧。你不需要遍历所有帧，只需要找到「状态变化的边界」。

## 游戏引擎中的实际应用

### UE5：每日签到最长连续天数

```cpp
// UE5 签到系统
int32 ULoginManager::GetLongestConsecutiveDays(
    const TArray<FDateTime>& LoginHistory)
{
    // 1. 将时间戳转为「一年中的第几天」
    TSet<int32> DaySet;
    for (const FDateTime& Time : LoginHistory)
    {
        DaySet.Add(Time.GetDayOfYear());
    }
    
    // 2. 找最长连续序列
    int32 MaxStreak = 0;
    for (int32 Day : DaySet)
    {
        if (DaySet.Contains(Day - 1)) continue; // 不是起点
        
        int32 End = Day;
        while (DaySet.Contains(End + 1)) End++;
        MaxStreak = FMath::Max(MaxStreak, End - Day + 1);
    }
    return MaxStreak;
}

// 使用：判断是否触发「连续 7 天签到」成就
void ULoginManager::CheckAchievement()
{
    int32 Streak = GetLongestConsecutiveDays(LoginHistory);
    if (Streak >= 7)
    {
        UnlockAchievement(TEXT("七日签到"));
    }
}
```

### Unity：连击评级链分析

```csharp
// Unity C# 战斗复盘系统
public class ComboAnalyzer : MonoBehaviour
{
    // 将连击过程中出现的所有评级转为 int，求最大连续跨度
    public int GetComboCoverage(HashSet<int> ratingsHit)
    {
        int maxCoverage = 0;
        foreach (int r in ratingsHit)
        {
            if (ratingsHit.Contains(r - 1)) continue;
            
            int end = r;
            while (ratingsHit.Contains(end + 1)) end++;
            maxCoverage = Mathf.Max(maxCoverage, end - r + 1);
        }
        return maxCoverage;
    }
    
    // 评级枚举
    enum ComboRank { D = 1, C = 2, B = 3, A = 4, S = 5, SS = 6, SSS = 7 }
}
```

### 为什么哈希集合在游戏开发中高频出现？

除了上一篇文章讲的哈希表 O(1) 查找，这道题还展示了哈希集合的另一个用途：**存在性快速判定**。游戏中大量逻辑只需要知道「有没有」，不需要「是什么」：

| 场景 | 数据结构 |
|---|---|
| 已解锁技能的 ID 集合 | `TSet<SkillID>` |
| 已探索地图区块 ID | `TSet<ChunkID>` |
| 已收集图鉴 ID | `HashSet<int>` |
| 当前离线好友列表 | `TSet<PlayerID>` |

这些场景都是「只需要 `Contains`，不需要 `operator[]`」——用集合而非映射，意图更清晰，内存更省。

## 变体与追问

### 追问一：数据量极大（n > 10⁷），内存放不下哈希集合？

→ **位图（Bitmap）**：如果数据范围有限（比如签到天数最多 365），用位图替代哈希集合，O(1) 查找不变，内存降到 365 bit ≈ 46 字节。

→ **外部排序 + 分块扫描**：如果数据范围也很大，可以先外部排序，再扫描，O(n log n) 但内存可控。

**游戏场景**：全年签到记录，365 天用位图是天然最优解。

### 追问二：要求返回序列本身，而不仅是长度？

→ 在计数过程中记录起点和终点，返回 `[start, end]`。

```cpp
pair<int, int> start, end;
// 在 while 循环中更新 end = cur
// ans = max(ans, cur - x + 1) 时同步更新 start = x, end = cur
```

### 追问三：如果有重复元素怎么办？

→ 哈希集合天然去重，不影响结果。但要注意：如果业务需要保留重复（比如同一天登录了两次），可以先做一层去重再套算法。

### 追问四：不止连续递增，还要求连续递减？

→ 完全对称——再跑一遍检查 `x+1` 不存在起点 → 向左数。或者把数组取反后直接复用原算法。

**游戏场景**：连击断档检测——战斗中从 SSS 一路掉到 D 的最长连续降级跨度。

### 面试高分回答模板

> 我做过游戏的每日签到系统。需求是计算玩家的最长连续登录天数，用来解锁「七日签到」类成就。
>
> 登录记录是时间戳数组，我先转成「一年中的第几天」，然后扔进 `TSet` 去重。
>
> 核心优化是：遍历集合时，如果当前天数 - 1 也存在于集合中，说明它不是连续段的起点，直接跳过。只有当它是起点时，才向后数到连续段结束。这样每个天数最多被访问两次，时间复杂度 O(n)。
>
> 还讨论了补签卡的影响——补签会插入额外的天数，可能改变起点/终点的判定，但算法逻辑不变，只是在数据预处理阶段把补签日期合并进去。

| 维度 | 核心 |
